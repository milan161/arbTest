# 011 API基础数据获取处理过程

## 1. 功能概述

LOF011模块负责从Woody API获取数据并生成基础数据文件（basic文件），主要功能包括：

- 从API获取基金数据
- 保存API数据到JSON和CSV文件
- 爬取外汇管理局汇率数据
- 生成和更新基础数据文件（GLD\_USO\_basic\_data.csv）
- 直接读取API返回的'date'字段作为精准的基准日期锚点

## 2. 数据流程

1. **API数据获取**：调用Woody API获取基金数据
2. **数据保存**：将API数据保存到`data/woodyAPI`目录，文件名格式为`Data_woodyAPI_YYYYMMDD_HHMM.json`和`Data_woodyAPI_YYYYMMDD_HHMM.csv`
3. **汇率数据爬取**：从外汇管理局网站爬取最新汇率数据
4. **日期锚定**：直接提取API数据中的 'date' 字段（基准日），彻底告别通过汇率盲猜日期的历史。如果本地 basic 表中没有该日期，则自动为其创建新行。
5. **数据填充**：将API数据填充到对应的日期行
6. **文件更新**：更新basic文件，保持历史数据完整

## 4. API字段读取与处理

### 4.1 主要API字段

| API字段         | 描述      | 处理方式             |
| ------------- | ------- | ---------------- |
| date          | 基准日日期 | **核心时间锚点**，直接决定数据写入 basic 表的哪一行 |
| netvalue      | 基金净值    | 用于填充SPY和QQQ的净值字段 |
| CNYholdings   | 基准日汇率   | 保留作为参考，**不再作为日期判断依据** |
| CNY           | 估值日汇率   | 用于参考，不作为主要日期判断依据 |
| symbol\_hedge | 对冲ETF信息 | 包含ETF价格和权重信息     |
| calibration   | LOF校准值  | 用于填充对应LOF基金的校准字段 |
| hedge         | 对冲值     | 用于计算LOF和ETF的数量对应 |

### 4.2 字段处理逻辑

## 5. 日期判断逻辑

### 5.1 汇率数据获取

- 从外汇管理局网站爬取最新的美元兑人民币中间价和发布日期
- 将汇率数据添加到basic文件中

### 5.2 API数据日期锚定

1. **解析 Date 字段**：遍历API数据，提取包含明确时间的 `date` 字段。
2. **日期格式统一**：将提取到的日期字符串安全解析并格式化为 `YYYY-MM-DD`。
3. **检查或创建新行**：在 `basic` 表中寻找该日期。如果不存在，则主动在表格顶部创建这一天的新行。
4. **严格对应填充**：无论哪一天运行程序，获取到的最新价格和仓位都将严格、安全地写入这个被明确锚定的 `date` 行中，杜绝历史账本覆写错误。

### 5.3 日期锚定核心代码

```python
for fund_code, fund_data in self.api_data.items():
    api_date_val = fund_data.get('date')
    if api_date_val and str(api_date_val).strip():
        try:
            # 解析并统一格式为 YYYY-MM-DD
            parsed_date = pd.to_datetime(str(api_date_val).strip()).strftime('%Y-%m-%d')
            api_data_date = parsed_date
            print(f"✅ 从 API 报文中成功提取到明确的基准日期: {api_data_date}")
            break
        except Exception as e:
            pass

# 如果这个基准日在 basic 表里还不存在，主动为它创建一行
if not combined_df['日期'].isin([api_data_date]).any():
    print(f"⚠️ 基础表中尚无 {api_data_date} 的记录，主动创建该行...")
    new_api_row = pd.DataFrame({'日期': [api_data_date]})
    combined_df = pd.concat([new_api_row, combined_df], ignore_index=True)
```

## 6. 数据填充逻辑

1. **ETF价格填充**：
   - 遍历API数据中的每个基金
   - 提取symbol\_hedge中的ETF价格
   - 去重处理，避免重复填充
   - 将价格填充到对应日期的对应列
2. **LOF校准值填充**：
   - 提取LOF基金的校准值
   - 填充到对应基金代码的校准列

## 7. 错误处理

- **文件占用**：如果basic文件被其他程序占用（如Excel），会尝试保存到临时文件
- **数据缺失**：如果API数据缺失某些字段，会保持对应列的值为None
- **汇率获取失败**：如果汇率爬取失败，会使用默认日期逻辑
- **日期格式不一致**：会自动将不同格式的日期统一转换为YYYY-MM-DD格式
- **API调用失败**：如果API调用失败，会自动使用最新的本地CSV数据，确保程序正常运行
- **API访问控制**：基于状态文件的时间区间控制，每个时间区间最多访问API一次，避免短时间内重复请求导致的服务器拒绝问题

## 8. API访问控制

### 8.1 时间区间划分

- **0:00 - 9:15**：此时API数据可能是昨天的旧数据
- **9:15 - 14:35**：9:15后API数据更新，可访问API
- **14:35 - 16:05**：14:35后日本和欧洲市场数据更新，可访问API
- **16:05 - 23:59**：16:05后香港市场数据更新，可访问API

### 8.2 状态管理

- 使用统一的`access_status.json`文件记录API访问状态
- 每个时间区间最多访问API一次
- 新的一天会自动重置访问状态
- 状态文件包含：
  - 最后访问日期
  - 各时间区间的访问时间

### 8.3 访问控制逻辑

```python
# 检查是否需要重置状态（新的一天）
if status['last_access_date'] != today:
    # 新的一天，重置状态
    status['last_access_date'] = today
    status['access_times'] = {
        '915': None,
        '1435': None,
        '1605': None
    }
    save_api_access_status(status)

# 9:15 - 14:35
elif time_915 <= current_time < time_1435:
    # 第一次运行，访问API
    if not status['access_times']['915']:
        return True, "[API] 9:15-14:35区间首次运行，访问API", '915'
    else:
        return False, "[API] 9:15-14:35区间已访问过API，不再重复访问", None
```

## 9. 注意事项

- （woody第二版更新后）API数据包含净值（基准日）日期信息，本程序通过CNYholding价格与汇率对比来确定基准日日期
- 每次运行会保留所有历史数据，只更新对应日期的数据
- 历史数据的检查和补充由013模块负责
- 代码中使用了去重逻辑，避免重复处理和打印相同的ETF数据
- 输出信息清晰，包含API数据获取、汇率数据获取、日期匹配和数据填充的详细信息

## 9. 代码优化

3. **数据回显**：添加了详细的数据回显，让用户可以清楚地看到获取的数据
