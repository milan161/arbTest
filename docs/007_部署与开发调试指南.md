## 三、服务器部署（Linux）

### 3.1 环境准备

```bash
# 1. 安装 Python 3.11
sudo apt update
sudo apt install python3.11 python3.11-venv

# 2. 安装 Node.js 18
curl -fsSL https://deb.nodesource.com/setup_18.x | sudo -E bash -
sudo apt install -y nodejs

# 3. 创建项目目录
mkdir -p /opt/arbdashboard
cd /opt/arbdashboard
```

### 3.2 部署后端

```bash
# 1. 创建虚拟环境
python3.11 -m venv .venv
source .venv/bin/activate

# 2. 安装依赖
cd backend
pip install -r requirements.txt

# 3. 使用 systemd 管理
sudo tee /etc/systemd/system/arbdashboard.service << EOF
[Unit]
Description=ArbDashboard Backend
After=network.target

[Service]
Type=simple
User=ubuntu
WorkingDirectory=/opt/arbdashboard/backend
Environment="PATH=/opt/arbdashboard/.venv/bin"
ExecStart=/opt/arbdashboard/.venv/bin/python main.py

[Install]
WantedBy=multi-user.target
EOF

# 4. 启动服务
sudo systemctl daemon-reload
sudo systemctl enable arbdashboard
sudo systemctl start arbdashboard

# 5. 查看状态
sudo systemctl status arbdashboard
```

### 3.3 部署前端

```bash
# 1. 安装依赖
cd /opt/arbdashboard/frontend
npm install

# 2. 构建生产版本
npm run build

# 3. 使用 Nginx 提供静态文件
sudo tee /etc/nginx/sites-available/arbdashboard << EOF
server {
    listen 80;
    server_name your-domain.com;

    location / {
        root /opt/arbdashboard/frontend/dist;
        try_files \$uri \$uri/ /index.html;
    }

    location /api {
        proxy_pass http://localhost:8000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade \$http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host \$host;
        proxy_cache_bypass \$http_upgrade;
    }
}
EOF

# 4. 启用配置
sudo ln -s /etc/nginx/sites-available/arbdashboard /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl restart nginx
```

## 四、数据库备份

### 4.1 手动备份

### 4.2 自动备份脚本

```python
# backend/utils/backup_db.py
import shutil
from datetime import datetime
from pathlib import Path

DB_PATH = Path(r"D:\Study\arbTest\database\arb_master.db")
BACKUP_DIR = Path(r"D:\Study\arbTest\database\backups")

def backup_database():
    """备份数据库"""
    BACKUP_DIR.mkdir(parents=True, exist_ok=True)
    timestamp = datetime.now().strftime("%Y%m%d_%H%M%S")
    backup_path = BACKUP_DIR / f"arb_master_backup_{timestamp}.db"
    shutil.copy2(DB_PATH, backup_path)
    print(f"数据库已备份到: {backup_path}")

if __name__ == "__main__":
    backup_database()
```

### 4.3 定时备份（Windows 任务计划程序）

## 二、`start_dashboard.bat` 启动流程（含 Debug）

批处理文件执行以下步骤：

```
步骤0: 杀死旧的 8000 端口进程 → 等待 2 秒
步骤1: 启动后端（新窗口）
步骤2: 健康检查循环（每 2 秒检查一次，最多 30 秒）
步骤3: 后端就绪后，启动前端（新窗口）
步骤4: 3 秒后打开浏览器
```

## 三、后端启动内部时序

后端 `main.py` 的启动分为多个阶段：

### 3.1 模块加载阶段（~1.5 秒）

```
17:04:23,655  TQ 数据接口全局抢占初始化
17:04:23,656  使用 arbcore 目录、数据库路径
17:04:23,836  SSE 白银期货后台线程启动
17:04:24,186  Core modules imported successfully     ← 模块导入完成
17:04:24,209  TradeManager 挂载通达信
17:04:24,211  TradingService 就绪
17:04:24,211  MarketDataService 初始化（IB Reader 后台线程启动）
17:04:24,212  富途 Reader 初始化
17:04:24,305  检测前端 dist 目录
17:04:24,468  Starting ArbNext Backend lifespan...   ← FastAPI lifespan 开始
```

**注意**：uvicorn 在 lifespan 之前就已经绑定端口 8000 并开始监听。这就是为什么健康检查能在 `17:04:24.80` 通过（lifespan 还在 `17:04:24.470` 才开始）。

### 3.2 Lifespan 初始化阶段（~1 秒）

```
17:04:24,469  分时采样服务启动
17:04:24,469  自动交易引擎已禁用（安全模式）
17:04:24,470  启动时自动运行 011 数据更新任务（daily_updater 子进程）
17:04:24,490  011 任务已在后台启动
17:04:24,500  DashboardSnapshotService 启动 → 仅刷新【黄金原油】【QDII欧美】  ← 低优先级 TAB 延迟 120 秒
17:04:24,510  清晨刷新定时器注册
17:04:24,510  自动净值更新定时器注册
```

**DashboardSnapshotService 启动时序（V8.2+，2026-07-03 优化）：**

| 时刻 | 事件 | 说明 |
|------|------|------|
| 启动时 | 刷新【黄金原油】【QDII欧美】数据 | 高优先级 TAB 立即可用 |
| 启动 + 120 秒 | 刷新全量+低优先级分类（QDII亚洲/国内LOF/白银/现金管理） | 给 daily_updater 留足完成时间 |

**设计意图**：每日首次启动时，`daily_updater` 子进程（011 任务）需要执行完整流水线（Woody VPS 同步、汇率、期货、份额、静态估值等），大约耗时 20-40 秒。DashboardSnapshotService 延迟 120 秒再启动低优先级分类的指数数据获取，避免：

- 网络/CPU 资源被低优先级 API 请求抢占
- 大量 `[INDEX-TENCENT]`/`[INDEX-EASTMONEY]` 日志淹没 daily_updater 的关键输出
- 前端低优先级 TAB 的首次加载变慢（延迟后再加载，数据已就绪）

### 3.3 后台连接阶段（~30 秒，非阻塞）

这些在后台线程中运行，不阻塞启动：

```
17:04:26,214  富途连接失败（3次），已禁用
17:04:29,876  国金QMT连接失败
17:04:34,493  行情引擎启动，挂载数据源
17:04:34,600  通达信挂载成功
17:04:42,834  IB连接失败（3次），已禁用
17:04:47,877  国金QMT检测3次失败
17:04:57,963  银河QMT检测3次失败
17:04:57,963  新浪财经挂载成功
17:04:57,963  实时行情引擎已启动                  ← 全部数据源挂载完成
17:05:00,002  富途 OpenD 连接正常
```

**关键**：这些后台连接失败（IB、QMT、富途）不影响前端页面展示，只是对应数据源不可用。

### 3.4 VPS 数据同步机制（V8.2+，2026-07-03 优化）

在 `daily_updater` 的完整流水线中，`_try_sync_all_from_vps()` 负责从云端 VPS 增量同步历史数据文件：

```
[VPS] 正在扫描云端所有缺失的 woody 历史数据...
  → 遍历 VPS 上所有 woody_*.json 文件
  → 对每个文件：检查本地是否存在 & 是否已标记为已同步
  → 未标记 → 读取解析 → 标记 access_sync_status(sync_date, "woody_vps_sync")
```

**历史问题（2026-07-03 前）**：跳过检查使用 key `"woody_vps_sync"`，但实际标记用 `"woody_lof_batch"` → key 不匹配 → 每次启动都重复解析所有历史文件（约 26 天 × 20+ 基金 = 20 秒浪费）。

## 四、常见冲突风险点

### 4.1 ECONNREFUSED（前端连不上后端）

**原因**：前端启动时后端端口 8000 未开。

**触发条件**：
- 手动先启动前端再启动后端
- `start_dashboard.bat` 健康检查被跳过或失败后仍继续

**解决**：始终使用 `start_dashboard.bat` 启动，不要手动分别启动。

### 4.2 端口 8000 被占用

**现象**：后端启动失败，报 `Address already in use`。

**原因**：上次的后端进程未完全退出。

**解决**：`start_dashboard.bat` 会自动杀死占用 8000 端口的旧进程。如果是手动启动，需先手动杀进程：
```bash
netstat -ano | findstr :8000
taskkill /F /PID <进程ID>
```

### 4.3 后端启动超时（30 秒内未就绪）

**现象**：健康检查 15 次全部失败，批处理暂停。

**可能原因**：
- Python 虚拟环境路径错误
- `arbcore` 模块导入失败
- 数据库文件不存在或损坏
- 端口被其他程序占用

**排查**：查看 "ArbNext Backend" 窗口的错误输出。

### 4.4 通达信 DLL 冲突

**现象**：后端日志报 `RuntimeError: TQ 数据接口未正确初始化`。

**原因**：多个程序同时使用通达信 TQ 接口。

**解决**：Master-Slave 架构自动处理 —— 如果 `LOFarb`（程序1）已运行占用端口 5000，ArbDashboard 自动进入 Slave 模式（禁用本地通达信驱动）。

### 4.5 前端 Vite 代理报错但后端正常

**现象**：前端控制台显示 `[vite] http proxy error: ECONNREFUSED`，但后端日志显示正常。

**原因**：Vite v8 的代理中间件在后端完全就绪前发出请求。

**解决**：刷新页面即可，后续请求正常。这是 Vite v8 的已知行为，无法完全抑制。

#### 错误5：通达信 TQ 接口未正确初始化（ctypes 回调异常）

**现象**：
控制台终端疯狂打印如下报错并刷屏：
```
Exception ignored on calling ctypes callback function: <bound method tq._data_callback_transfer of <class 'tqcenter.tq'>>
Traceback (most recent call last):
  File "D:\new_tdx_test\PYPlugins\user\tqcenter.py", line 950, in _data_callback_transfer
    run_id = cls._get_run_id()
             ^^^^^^^^^^^^^^^^^
  File "D:\new_tdx_test\PYPlugins\user\tqcenter.py", line 571, in _get_run_id
    raise RuntimeError("TQ数据接口未正确初始化")
RuntimeError: TQ数据接口未正确初始化
```

**根因分析**：
`tqcenter` 底层是 C/C++ 封装的接口，调用 `tq.initialize()` 时强制要求传入通达信插件安装目录下 `tqcenter.py` 的绝对物理路径。若使用了 `tq.initialize(__file__)` 或其他非通达信目录路径，将触发底层完整性校验失败。

**关键细节**：`_data_callback_transfer` 是注册在 C 层的 **ctypes 回调**，它调用 `cls._get_run_id()` 是通过 **classmethod 描述符协议**。因此普通的 `tq._get_run_id = safe_func` 只 patch 了实例属性，**无法拦截** `cls._get_run_id()` 的调用。必须同时 patch 类字典：`type(tq)._get_run_id = staticmethod(safe_func)`。

**修复策略（多层防御，已在 `main.py` 实现）**：

1. **初始化路径修正**：将所有 `tq.initialize(__file__)` 修正为正确路径：
   ```python
   tdx_plugin_path = os.path.join(tdx_api_path, 'tqcenter.py')
   tq.initialize(tdx_plugin_path)
   ```

2. **拦截重复初始化**：用 `safe_initialize()` 替换 `tq.initialize`，防止后续模块重复初始化导致路径覆盖。

3. **Patch `_get_run_id`（实例+类双层）**：
   ```python
   tq._get_run_id = safe_get_run_id           # 实例层 patch
   type(tq)._get_run_id = staticmethod(safe_get_run_id)  # 类字典 patch
   ```

4. **Patch `_data_callback_transfer` 回调本体**：彻底在回调入口拦截 RuntimeError，且只记录一次日志，避免刷屏：
   ```python
   _orig_callback = getattr(tq, '_data_callback_transfer', None)
   tq._data_callback_transfer = safe_data_callback_transfer
   type(tq)._data_callback_transfer = safe_data_callback_transfer
   ```

**注意**：如果以上 Python 层 patch 仍无法拦截（因为 ctypes 回调可能已在 DLL 加载时绑定了原始 C 函数指针），则需确保通达信客户端已启动并正常运行。

#### 错误5：通达信 TQ 接口未正确初始化（ctypes 回调异常）

**现象**：
当启动程序3且通达信客户端未打开时，控制台终端疯狂打印如下报错并刷屏：
```
Exception ignored on calling ctypes callback function: <bound method tq._data_callback_transfer of <class 'tqcenter.tq'>>
Traceback (most recent call last):
  File "D:\new_tdx_test\PYPlugins\user\tqcenter.py", line 950, in _data_callback_transfer
    run_id = cls._get_run_id()
             ^^^^^^^^^^^^^^^^^
  File "D:\new_tdx_test\PYPlugins\user\tqcenter.py", line 571, in _get_run_id
    raise RuntimeError("TQ数据接口未正确初始化")
RuntimeError: TQ数据接口未正确初始化
```

**底层技术背景与知识科普**：

1. **什么是通达信 Python 插件架构（沙箱运行 vs. 外部运行）**：
   - 通达信的 `tqcenter` 插件底层主要是用 C/C++ 动态链接库（DLL）编写的。
   - 当你在通达信软件内运行脚本时，Python 解释器是内嵌在通达信软件内部的，环境参数完全自动传递。
   - 当你在 CMD、PowerShell 等外部环境中独立启动程序3（FastAPI 网页服务）时，你脱离了通达信的沙箱环境。此时，`tqcenter.py` 底层的 C 库需要通过文件系统和特定绝对路径，反向搜寻并连接本地正在运行的通达信客户端程序。

2. **为什么普通的 Python 猴子补丁（Monkey Patch）会失效？**：
   - **C 语言 ctypes 回调机制**：`tqcenter.py` 底层的 DLL 驱动会将回调函数（例如 `_data_callback_transfer`）直接注册到 C/C++ 内存的底层回调链上。
   - 当 DLL 内部状态发生改变或重连时，DLL 会直接跳转到最初绑定该函数时的 **C 语言指针**。
   - 这意味着，如果你在 Python 中简单地写 `tq._data_callback_transfer = new_func`，DLL 的 C 层根本无法感知这个 Python 层的属性替换，它依然会调用原始注册的 `_data_callback_transfer`！
   - 同理，`_get_run_id` 是一个被 `@classmethod` 修饰的类方法。如果在回调中使用了 `cls._get_run_id()` 形式调用，它会直接通过类字典描述符协议获取方法。普通的实例 patch `tq._get_run_id = safe_func` 会因为被类字典劫持而**完全失效**！

3. **双重 Patch 深度拦截方案**：
   为了防止在通达信没有登录时 DLL 依然在后台死循环尝试重连并疯狂刷屏报错，系统在 `main.py` 中部署了深层拦截：
   - **类字典拦截**：利用 `type(tq)._get_run_id = staticmethod(safe_get_run_id)` 绕过类属性查找机制，强制所有的 `cls._get_run_id()` 回归到安全代理。
   - **回调外衣包裹**：在 C 库尚未发起回调注册前，将 `tq._data_callback_transfer` 重新替换为带有异常捕捉（`try-except`）的安全代理函数，在抛出第一个 `RuntimeError` 时打印提示，之后所有报错直接静默，防止控制台刷屏。

#### 错误6：未检测到已就绪的通达信(TDX)交易账户（启动熔断退出）

**现象**：
双击运行 `start_dashboard.bat` 启动程序3时，后端终端（ArbNext Backend 窗口）输出一个极其显眼的红字警告框，随后后端进程主动崩溃退出（因为 `start_dashboard.bat` 使用了 `cmd /k`，所以命令行窗口会保留在桌面供你阅读报错）：
```
================================================================================
🚨🚨🚨 致命错误：未检测到已成功连接的【通达信】交易账户！ 🚨🚨🚨
================================================================================
👉 原因分析：
   1. 本地【通达信客户端 (TdxW.exe)】尚未启动，或未成功登录交易账号。
   2. tqcenter 插件无法取得有效的 stock_account 句柄。
...
```

**设计意图与主从隔离逻辑**：

为了让程序3彻底在本地挑起交易重任，并能随时安全稳定地调用通达信，系统采用了**严密的启动熔断机制**：

1. **主从模式判定**：
   - 看板在启动时，会尝试探测本地 `5000` 端口。如果发现程序1 (LOFarb) 正在运行，它将自动进入 **从机模式 (Slave)**，禁用自身的通达信接口，完全依靠程序1去同步数据和处理交易，防止两个程序同时写入通达信 DLL 发生文件锁死或冲突崩溃。此时程序3能顺利开启。
2. **独立运行检验（启动熔断）**：
   - 如果程序1没有运行，说明程序3将被当作**核心交易主程序**独立运行。
   - 此时，程序3会非常严格地检测通达信连接：它会在 `sys.exit(1)` 前通过 `tq.stock_account()` 获取本地交易账号句柄。
   - 如果你的通达信客户端没有启动，或者尚未成功登录（即获取到的账号句柄为 `None` 或 `0`），程序3会认定这属于**交易风险环境**。它不会在后台静默失败，而是立刻弹出上方的**巨幅警告框**，并强行中断并退出，阻止不安全的后台启动。

**日常排查及正确启动顺序**：
1. **第一步**：先双击打开你电脑上的【通达信交易客户端】，输入账号密码并**成功登录**进去（保持通达信在后台运行）。
2. **第二步**：双击运行主目录下的 `start_dashboard.bat`。此时，后端程序会瞬间检测到通达信的 stock_account 句柄，绿灯放行，正常启动看板！

#### 问题3：自动交易引擎的“开启/暂停”状态代表什么？

**解答**：
*   **引擎指代**：这指的是后台运行的自动网格盯盘与自动交易执行引擎（`AutoTradeRunner`）。
*   **状态说明**：
    *   **开启状态**：系统会实时扫描所有激活的策略规则（折溢价阈值等），一旦触发买卖信号且启用了真实报单，系统将**自动调用底层交易通道（通达信/QMT）执行真实下单委托**。
    *   **暂停状态**：系统处于挂起休眠状态，**不会执行任何自动下单或交易动作**，此时系统仅用于只读展示，以保障实盘安全。您随时可以在策略看板或右上角手动一键切换。
*   **物理隔离设计的意义**：数据抓取是高频且只读的，而交易下单具有实盘资金风险。物理隔离设计可以保证：即使行情数据如何跳动，只要自动交易引擎关闭，就绝不会发生任何实盘误交易，保障系统在不交易时的绝对安全。

#### 问题4：顶部行情状态中“美港: IB/富途”代表什么？两个都开启时程序到底用哪个？

**解答**：
*   **首选主通道（IB）**：当 IB 和富途都开启并成功连通时，顶部显示为 `美港: IB/富途`。此时，系统会把**所有美股行情请求发送给 IB**，因为它是您的主数据源。富途此时处于 **“热备份（Hot Standby）”** 状态，在线但静默，以节省富途 API 请求频次。
*   **无缝备用切换（富途）**：一旦 IB 客户端被关闭、断开连接，或者在获取价格时发生超时报错，系统会**瞬间、无缝地将行情抓取请求切换到富途**，由富途接管，确保您的实时估值和折溢价套利计算绝不断网。
*   **具体显示规则**：
    *   若只显示 `美港: IB`，说明只有 IB 在线，富途 OpenD 未开启。
    *   若只显示 `美港: 富途`，说明 IB 未开启或已断开，系统已自动降级至富途通道读取数据。
    *   若两者均未开启，系统显示 `美港: 未连接`。

## 十、前端筛选逻辑Bug修复（2026-06-10）

### 10.1 问题：选择分类后页面显示空白

**现象**：
用户在"全场套利机会实时雷达"页面选择"黄金原油"分类，并设定折价率阈值范围后，页面显示空白，没有任何基金数据。


前端复选框使用的是**英文key**：
```javascript

```

**错误的筛选逻辑**（`frontend/src/views/Analysis.vue` 第853行）：
```javascript
// ❌ 错误：直接比较英文key和中文category，永远不匹配
data = data.filter((f: any) => selectedCategories.value.includes(f.category))
```

**Bug #2：折价率阈值筛选逻辑错误**

**错误的筛选逻辑**（第859行）：
```javascript

data = data.filter((f: any) => f.rt_premium >= lower && f.rt_premium <= upper)
```

**问题**：这筛选的是折价率在 -0.5% 到 2.0% **之间**的基金，但折价套利的需求是：
- **折价越大越好**：折价率 ≤ 开仓阈值（如 ≤ -0.5%）→ 低价买入机会
- **溢价过高要平仓**：溢价率 ≥ 平仓阈值（如 ≥ 2.0%）→ 利润空间小，平仓

**1. 添加分类映射表**（在组件script部分定义）：

```javascript
  'gold_oil': ['黄金原油'],
  'qdii_us': ['纯ETF', 'QDII欧美', '混合跨境', '指数'],
  'qdii_asia': ['QDII 亚洲'],

```

**2. 修改筛选逻辑**：
```javascript

    const res = await axios.get('/api/dashboard')
    let data = res.data.data
    
    // 分类筛选（使用映射表）
    if (selectedCategories.value && selectedCategories.value.length > 0) {
      // 收集所有选中的分类对应的数据库category值
      const allowedCategories = new Set<string>()
      for (const catKey of selectedCategories.value) {
        const mappedCats = categoryMap[catKey] || []
        for (const cat of mappedCats) {
          allowedCategories.add(cat)
        }
      }
      data = data.filter((f: any) => allowedCategories.has(f.category))

    opportunityData.value = data

}
```

## 十一、美股实时订阅白名单机制（解决 IB/富途 频控与刷屏）

### 9.1 设计背景
随着系统接入的跨境/混合基金越来越多（如 501312, 160644, 501225 等），这些基金的估值组合中引入了大量海外 ETF 成分股。
原系统会自动遍历所有基金下的成分股，并一股脑全部向盈透证券 (IB) 或富途发起实时订阅和看门狗历史数据拉取。
由于很多成分股并没有美股夜盘（Overnight）报价且我们并不需要对其进行对冲，IB 接口因为频繁返回空数据和发送超限，频繁报出 Pacing Violation (频控违规) 错误并使终端疯狂刷屏。

### 9.2 解决方案：自定义白名单
系统在数据中控台增加了一个**“美股订阅白名单 (IB/富途)”**配置功能，实现了以下架构逻辑：

1. **白名单数据库存储**：
   - 数据存储在 `data_source_config` 表中，标识为 `module = 'ib_config'` 且 `source_name = 'whitelist'`。
   - 配置字段 `config_json` 存储 JSON 格式的名单列表：`{"symbols": ["GLD", "USO", "XOP", "SLV", "SPY", "QQQ", "INDA"]}`。

2. **接口与内存同步**：
   - 后端提供了 `GET /api/config/ib_symbols` 与 `POST /api/config/ib_symbols/update` 两个 REST 接口用于读取与更新。
   - 当用户在前端保存了新的白名单后，后端会**即时更新内存中运行的 `ib_reader.symbols` 对象**。这意味着修改白名单**不需要重启后端服务**，更新瞬间在长连接中生效！

3. **读取优先级逻辑（在 `ib_reader.py` 轮询中实现）**：
   - `ib_reader.py` 后台轮询线程每次循环都会优先检测数据库中是否存在该白名单配置。
   - **如果存在白名单**：直接加载白名单中的代码作为订阅池，其他成分股将被**完全跳过**，不再向 IB 接口发起任何请求。
   - **如果留空（无白名单）**：系统会降级到原有的 YAML 提取逻辑，自动抓取 `lof_config.yaml` 中所有的底层美股。

### 9.3 前端使用指南
1. 打开左侧菜单的“数据管理”页面（Data.vue）。
2. 在左侧的**“脚本任务中控台”**正下方，找到“💡 美股订阅白名单 (IB/富途)”输入框。
3. 输入你需要关注的重点美股，用英文逗号分隔（例如：`GLD, USO, XOP, SLV, SPY, QQQ, INDA`）。
4. 点击**“保存白名单”**，前端会调用后端接口存入 SQLite。
5. 后端接收后，在控制台会即时输出 `🔄 已动态向 IBReader 推送新订阅白名单: [...]`。此时，系统立即对非白名单品种断流，彻底杜绝多余的 IB 请求！
