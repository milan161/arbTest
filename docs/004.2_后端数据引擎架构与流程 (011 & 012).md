# 后端数据引擎架构与流程 (011 & 012)

## 1. 架构总览：双轨制数据流水线

LOF 基金套利系统的后端数据抓取与计算，由 `011` 和 `012` 两个核心模块协同完成。为了保证极高的数据稳定性和防断流能力，系统采用了**“官方API为主 + 网页爬虫兜底”**的双轨制架构。

- **`011` (数据骨架与 API 搬运工)**：纯粹、轻量。只负责确定日期锚点（外汇管理局），并从 Woody 官方 API 接口极速搬运底层锚点数据，生成规整的 `basic` 基础表。
- **`012` (血肉缝补与终极结算师)**：庞大、精密。负责检查 `011` 生成的表格是否有空洞，动用新浪和 Woody 爬虫强力填补所有缺失的美股、期货数据。最后拉取 A 股实时净值，完成庞大的静态估值运算。

---

## 2. 核心账本：GLD_USO_basic_data.csv 数据拼图

`GLD_USO_basic_data.csv` 是整个系统的“基石账本”。它固定包含 25+ 列数据，每一列都有极其严格的“血统”和获取途径。

### 🧩 A. 由 `011` (API/官方) 负责写入的主干数据
*   **`日期` & `人民币中间价`**：来自外汇管理局官方接口。这也是确定 T 日和 T-1 日的核心时间锚点。
*   **API 核心基准价 (`GLD_price`, `USO_price` 及变种权重等)**：直接从 Woody API 的 JSON 数据打平铺开写入。
*   **LOF 物理兑换比 (`162411校准/对冲` 等)**：直接从 Woody API 解析并写入。

### 🧩 B. 由 `012` (爬虫兜底) 负责填补的血肉数据
当 `011` 执行完毕后，`012` 会扫描表格，如果发现数据空缺，将启动以下两套爬虫引擎进行缝补：

**引擎 1：Woody 专属网页爬虫 (`WoodyWebCrawler`)**
*   **兜底目标**：大宗商品GLD、USO及区域变种。
*   **抓取内容**：`GLD`, `USO` 本尊，以及新浪无法获取的欧洲/日本变种 (如 `^GLD-EU`, `^USO-JP` 等)。
*   **特殊抓取**：强行解析网页右上角，获取 `黄金校准` 和 `原油校准`。

**引擎 2：新浪极速流 (`Sina API`)**
*   **兜底目标**：标准美股 ETF、宏观指数与期货。
*   **抓取内容**：
    *   **标准美股 ETF**：`XOP`, `SPY`, `QQQ`, `SLV`, `XBI` 等（带 `价格` 后缀与 Woody 数据物理隔离）。
    *   **宏观指数**：标普 500 (`.INX`)，纳斯达克 100 (`.NDX`)。
    *   **期货结算价**：黄金 (`GC_settle`)，原油 (`CL_settle`)，纳指微型 (`NQ_settle`)，标普微型 (`ES_settle`)。

## 3. 011 执行流程详解 (API 请求器)

`LOF011_generate_basic_data.py` 是一个纯净的、轻量级的预处理脚本。

1.  **取时间锚点**：访问国家外汇管理局，获取当日 `人民币中间价` 及发布日期。
2.  **判断交易日**：根据 A 股节假日日历，推算出需要计算估值的 T 日 和 T-1 日。
3.  **访问 Woody API**：
    *   根据配置文件 `lof_config.yaml` 提取所有需要监控的 LOF 代码。
    *   调用 API，获取返回的庞大 JSON 字典。
    *   将 JSON 数据进行**扁平化处理 (Flattening)**，将嵌套的 `symbol_hedge` 字典拆解为固定的 `_price` 和 `_ratio` 字段。
4.  **智能 YAML 反哺**：如果 API 数据中包含新的仓位 (`position`) 或 ETF 权重，程序会自动反向修改本地的 `lof_config.yaml` 配置文件，保持策略的绝对同步。
5.  **生成基石 CSV**：将扁平化的数据存入带时间戳的 `Data_woodyAPI_XXX.csv`，最后整合出骨架版的 `GLD_USO_basic_data.csv`。

## 4. 012 执行流程详解 (缝补与终极结算)

`LOF012_generate_lof_data.py` 是整个后端计算的“心脏”。

1.  **加载骨架数据**：读取 011 准备好的 `basic` 表格。
2.  **兜底缝补阶段** (见第 2 节的拼图说明)：
    *   检查 `黄金校准` 等数据，缺失则呼叫 `WoodyWebCrawler` 爬网页补齐。
    *   检查期货结算价，缺失则呼叫新浪接口补齐。
    *   检查区域变种 ETF (`^GLD-EU` 等) 和基础商品本尊 (`GLD`/`USO`)，缺失则让 Woody 爬取其历史净值填补。
    *   检查标准美股 ETF 和指数 (`SPY`, `XOP`, `.INX` 等)，缺失则让新浪闪电爬取补齐。
3.  **获取 A 股本尊数据**：
    *   从东方财富网爬取各个 LOF 的**历史官方净值**。
    *   从新浪爬取各个 LOF 的**历史收盘价**。
4.  **大合并与庞大计算 (Outer Merge)**：
    *   将补齐血肉的 `basic` 宏观表，与 A 股 LOF 历史表按日期进行对其大合并。
    *   根据 `lof_config.yaml` 中配置的公式（双轨制）：
        *   计算 **ETF静态估值** 及其误差、溢价率。
        *   计算 **期货静态估值** (含微型期货 `MES`, `MGC` 等) 及其误差、溢价率。
        *   针对指数基金单独计算 **纯指数静态估值**。
5.  **落库**：将每只 LOF 基金独立拆分，生成供前端沙盘极速读取的 `LOF_160719_history.csv` 等文件。

## 5. 防封禁与时序安全控制

由于频繁请求第三方 API 和网页极易导致 IP 被封，系统设计了精密的防护策略。

### 5.1 状态隔离机制 (`access_status.json`)
系统会在 `data/access_status.json` 中永久记录 API 的访问时间与成功状态，防止任何重试逻辑引起无限死循环。

### 5.2 时段控制策略 (针对 Woody API)
针对 Woody 网站的发布规律，强制划分时间段：
- **0:00 - 9:15**：数据尚未更新，拒绝请求。
- **9:15 - 14:35**：A 股开盘前，只请求 1 次。
- **14:35 - 16:05**：日本等亚盘数据更新，允许请求 1 次。
- **16:05 - 23:59**：欧洲等盘口更新，允许请求 1 次。

### 5.3 数据驱动的静默策略 (新浪/东财爬虫)
对于新浪和东方财富的数据获取，完全抛弃“每天定时爬”的老旧思路，采用**数据驱动**：
程序在抓取前会先看本地的 `LOF_XXX_history.csv`，如果里面已经有了 T-1 日的收盘价和净值，则直接静默跳过网络请求，实现 0 延迟启动。只有当数据发生真实断层时，才会精准发起网络爬虫索要缺失的那一天数据。

## 6. 异常应对指南

| 异常现象 | 排查思路 |
|---|---|
| **控制台卡在 `[Woody补漏]`** | Woody 网站可能正在维护或触发了防爬拦截。程序会在超时后自动降级跳过，如果长时间卡死，可以直接关闭黑窗口重新运行。 |
| **保存 CSV 报错 `PermissionError`** | 表格正在被 Excel 打开。程序已内置防护，会暂时保存为 `.tmp` 文件，您只需关闭 Excel 重新运行脚本即可。 |
| **某天没有计算出估值误差** | 该天的 `basic` 宏观表中大概率缺失了某一个基准 ETF 的价格。通常是因为遇上了中外节假日不对齐导致的非交易日。 |
| **新增了一只纳指 LOF，沙盘报错** | 请确保在 `lof_config.yaml` 中配置了正确的 `trade_etf` 和 `trade_future`，`012` 的新浪兜底引擎会自动开始为其爬取数据。 |
