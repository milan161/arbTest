## 一、 基金配置文件 `lof_config.yaml` 的三大核心字段职能

`lof_config.yaml` 是整个套利系统的唯一核心配置源，控制着每只监控基金在系统中的运行方式。每只基金的配置围绕以下三大职能展开：

### 1.1 分流职能
*   `category` (String): 主分类 TAB。决定基金显示在前端主看板的哪个 TAB（如：黄金原油、QDII欧美、QDII日本、QDII亚洲、国内LOF、现金管理、白银）。
*   `sub_category` (String): 子分类。控制估值引擎对具体计算公式的选择（如：原油、黄金、纯ETF、指数、其他）。

### 1.2 估值职能
*   `woody_supported` (Boolean): 是否支持 Woody API。
*   `position` (Float) / `holdings.equity_ratio` (Float): 底仓仓位百分比（如 `88.45`），用于计算现金与持仓的配比。
*   `valuation_portfolio` (List): 估值一篮子底层组件及理论权重分配。
*   `related_index` (String): 关联指数代码，在 Woody 数据不支持时用作估值计算的参考基准。
*   `rate_type` (String, 可选): 若为 `spot`，表示实时估值必须取新浪 USD/CNY 实时在岸汇率；若缺省，则取央行中间价（`usd_cny_mid`）。

### 1.3 对冲职能
*   `trade_etf` (String): 套利交易对应的 ETF 现货代码（如 `USO`, `GLD`, `XOP`）。
*   `trade_future` (String): 套利对冲对应的主力期货合约代号（如 `MCL`, `GC`, `CL`）。
*   `future_hedging` (List): 期货对冲详情。包含期货标的、权重、到期月份等。*(注：目前对冲主要采用 ETF 现货，期货对冲尚未完全成熟，之前旧版设计的 Beta 系数调节逻辑已被废弃)*
    ```yaml
    future_hedging:
      - symbol: MCL            # 微原油期货
        delivery_month: '2604'   # 合约月份
        weight: 100.0          # 期货权重
        currency: USD
        exchange: SMART
    ```

## 二、 核心基金分类之“估值与对冲”算法详解

本套利系统最核心的价值在于“黄金原油”、“QDII欧美”、“QDII日本”三大类基金的估值和对冲实现。

### 2.1 黄金原油 (Woody API 支持，大宗商品跟踪)

#### 2.1.1 估值算法：矩阵公式 (Basket Valuation)
*   **适用条件**：黄金原油基金（`fund_basket_weights` 有记录或 YAML 中配置有多只估值标的，如南方原油 501018）。
*   **数学公式**：
    $$\text{实时估值} = \text{NAV}_{T-1} \times \left(1.0 + \text{position} \times \left( \left(\sum_{i} \text{price\_ratio}_i \times \text{weight}_i \right) \times \text{fx\_ratio} - 1.0\right)\right)$$
    *   $\text{price\_ratio}_i = \frac{\text{当前实时价}_i}{\text{基准日收盘价}_i}$
    *   $\text{fx\_ratio} = \frac{\text{当前汇率}}{\text{基准日汇率}}$
*   **锚点 (Anchor) 机制与时差处理**：
    为了应对全球市场时差，估值篮子中包含带区域后缀的锚点（如 `^USO-EU`, `^USO-JP`, `^USO-HK`）。
    *   **分母部分（基准日收盘价）**：使用带后缀符号（如 `^USO-EU`）查询数据库 `usa_etf_daily_prices` 对应的历史收盘价格。
    *   **分子部分（当前实时价）**：去掉后缀和 `^` 前缀，直接获取美股 `USO` 的当前最新成交价进行实时计算。
    *   **多空头寸保留**：支持负权重，限制过滤时必须使用 `weight != 0`，不得使用 `weight > 0` 过滤，以保留对冲和空头头寸。

#### 2.1.2 对冲算法
*   **对冲工具选择**：交易 YAML 配置中的 `trade_etf` (如 USO, GLD) 现货，或根据 `future_hedging` 中的期货参数（如 MCL 微原油期货）对冲。
*   **对冲数量换算**：目前主流采用等市值换算（即套利持仓市值除以期货合约乘数与价格）。*(纯期货校准及估值算法待进一步研发)*

---

### 2.2 QDII欧美 (Woody API 支持，欧美 ETF/指数跟踪)

#### 2.2.1 估值算法：魔法公式 (Magic Valuation)
*   **适用条件**：单一 ETF 欧美基金，且 Woody 提供了对冲因子 `hedge`（如华宝油气 162411 对应 XOP）。
*   **数学公式**：
    $$\text{实时估值} = \text{NAV}_{T-1} \times (1.0 - \text{position}) + \frac{\text{ETF当前实时价} \times \text{当前汇率}}{\text{hedge}}$$
    *   $\text{hedge}$ 反映基金净值与美股标的价格的换算关系（由 Woody API 每日更替，反映基金经理的对冲持仓占比）。
    *   **在岸价汇率机制 (`spot`)**：对配置了 `rate_type: spot` 的在岸价基金（如 `513350` 富国标普石油、`159518` 嘉实标普石油、`159502` 嘉实标普生物），其“当前汇率”严格取盘中新浪 USD/CNY 实时在岸价。当在岸价接口超时失效时，跳过估值以防止算错。其他普通基金的当前汇率则取央行中间价。

#### 2.2.2 对冲算法
*   **对冲工具选择**：通常直接交易美股对应 ETF（如 XOP, XBI, XLY）进行日内对冲，无期货合约对冲。

---

### 2.3 QDII日本 (无 Woody 支持，跟踪日经225)

#### 2.3.1 估值算法：指数公式 (Index Valuation)
*   **适用条件**：无 Woody 数据的日经 225 基金（513000、513520、159866）。
*   **数学公式**：
    $$\text{实时估值} = \text{NAV}_{T-1} \times \left(1.0 + \text{position} \times \left( \text{idx\_ratio} \times \text{fx\_ratio} - 1.0\right)\right)$$
    *   $\text{idx\_ratio} = \frac{\text{日经225当前实时值}}{\text{基准日日经225收盘值}}$
    *   $\text{fx\_ratio} = \frac{\text{日元当前汇率 (JPY/CNY)}}{\text{基准日日元汇率}}$
    *   $\text{position}$ 默认取 `pos_ratio` 的配置值（默认 0.95）。

#### 2.3.2 对冲算法
*   **对冲工具选择**：属于指数型 QDII，通常不进行日内对冲，主要进行场内折溢价申赎套利。

## 三、 关键边缘逻辑

### 3.1 溢价率计算逻辑 (T-1 净值)
为了防止净值更新时引入时效重影，静态/实时溢价率计算**必须且只能使用前一日（T-1）净值**：
$$\text{溢价率} = \left(\frac{\text{价格}}{\text{NAV}_{T-1}} - 1.0\right) \times 100$$
盘后静态估值（daily_updater）计算好 `static_val` 和 `valuation_error` 后直接写入 `unified_fund_history`，查询时禁止动态重新计算。

### 3.2 美国假期与时差回溯机制
针对投资美国市场的基金，系统在 `dynamic_valuation.py` 中内置了假期回溯算法。
当最新净值日 $T-1$ 恰逢美国假期（即 `usa_etf_daily_prices` 在此日期无 ETF 价格）时，系统会自动向前回溯寻找最近一个**既有净值、又有美股 ETF 历史收盘价**的有效交易日作为基准日。

## 1. 估值核心因子的数学本质

系统中进行盘中动态估值和盘后静态计算主要依赖两个关键因子：**校准因子 (Calibration)** 和 **对冲因子 (Hedge)**。这两个因子均来自每日 Woody API 数据同步。

### 1.1 校准因子 (Calibration)
*   **数学定义**：
    $$\text{Calibration} = \frac{\text{基准ETF价格}_{T-1} \times \text{汇率}_{T-1}}{\text{净值}_{T-1}}$$
*   **物理意义**：代表在 $T-1$ 日（基准日），每 $1$ 份 LOF 基金净值在理论上等价于多少人民币价值的底层 ETF 组件。

### 1.2 对冲因子 (Hedge)
*   **数学定义**：
    $$\text{Hedge} = \frac{\text{Calibration}}{\text{position}} = \frac{\text{基准ETF价格}_{T-1} \times \text{汇率}_{T-1}}{\text{净值}_{T-1} \times \text{position}}$$
*   **物理意义**：代表大约需要多少份底层标的 ETF，才能完美对冲 $1$ 份基金净值（结合了仓位因素）。
*   **常量折叠**：Woody API 会根据基金经理每日的真实仓位，在后台计算出这一比率并直接传递。通过引入 $\text{Hedge}$，系统能够将原本复杂的篮子权重计算折叠为 $O(1)$ 的简单算术，极大地提高了高频实时估值的计算效率。

## 2. 核心估值公式的实现与数学证明

根据基金的不同种类与因子的缺失状态，系统分别调用以下核心估值公式：

### 2.1 魔法公式 (Magic Valuation)
*   **适用条件**：单一 ETF 欧美基金且 Woody 提供了对冲因子 `hedge`（如华宝油气 162411 对应 XOP）。
*   **公式实现**：
    $$\text{估值} = \text{NAV}_{T-1} \times (1.0 - \text{position}) + \frac{\text{ETF实时价}_T \times \text{汇率}_T}{\text{Hedge}}$$
*   **物理等价性证明**：
    魔法公式将基金净值拆分为**非持仓现金部分**（用上一交易日真实净值计价，即 $\text{NAV}_{T-1} \times (1 - \text{pos})$）与**持仓部分**（按标的实时价格和实时汇率换算）。
    将 $\text{Hedge}$ 的定义带入上式：
    $$\text{估值} = \text{NAV}_{T-1} \times (1.0 - \text{position}) + \frac{\text{ETF实时价}_T \times \text{汇率}_T}{\frac{\text{ETF价格}_{T-1} \times \text{汇率}_{T-1}}{\text{NAV}_{T-1} \times \text{position}}}$$
    进一步化简：
    $$\text{估值} = \text{NAV}_{T-1} \times (1.0 - \text{position}) + \text{NAV}_{T-1} \times \text{position} \times \left( \frac{\text{ETF实时价}_T}{\text{ETF价格}_{T-1}} \times \frac{\text{汇率}_T}{\text{汇率}_{T-1}} \right)$$
    提取公因式 $\text{NAV}_{T-1}$：
    $$\text{估值} = \text{NAV}_{T-1} \times \left( 1.0 + \text{position} \times \left( \frac{\text{ETF实时价}_T}{\text{ETF价格}_{T-1}} \times \frac{\text{汇率}_T}{\text{汇率}_{T-1}} - 1.0 \right) \right)$$
    证明完毕。该式在数学上完全等价于**以基准日净值为锚点，乘以持仓资产在日内的价格和汇率变动幅度**。

### 2.2 矩阵公式 (Basket Valuation)
*   **适用条件**：黄金原油等拥有多资产持仓篮子的基金（`fund_basket_weights` 有多条权重记录，或魔法公式因缺少 hedge 因子降级时）。
*   **公式实现**：
    $$\text{估值} = \text{NAV}_{T-1} \times \left( 1.0 + \text{position} \times \left( \left(\sum_{i} \frac{\text{ETF实时价}_{T, i}}{\text{ETF价格}_{T-1, i}} \times \text{weight}_i\right) \times \frac{\text{汇率}_T}{\text{汇率}_{T-1}} - 1.0 \right) \right)$$
*   **多空头寸与负权重支持**：当篮子持仓中包含负权重（空头头寸）时，传统的 `weight > 0` 过滤规则会将负权重的对冲成份误杀剔除。因此，系统算法严格使用 `weight != 0` 过滤，以保留对冲和反向头寸，确保矩阵计算在多空市场下的准确性。

### 2.3 指数公式 (Index Valuation)
*   **适用条件**：无 Woody 数据的日本基金（513000、513520、159866）。
*   **公式实现**：
    $$\text{估值} = \text{NAV}_{T-1} \times \left( 1.0 + \text{position} \times \left( \frac{\text{实时指数}_T}{\text{基准指数}_{T-1}} \times \frac{\text{日元汇率}_T}{\text{日元汇率}_{T-1}} - 1.0 \right) \right)$$
    *   $\text{position}$ 默认取 `pos_ratio`（默认为 95%）。

## 3. 实时估值中的汇率控制机制

系统支持根据基金的不同特性，采用不同的实时汇率数据源以最大化拟合基金净值的真实涨跌。

### 3.1 央行中间价与实时在岸价 (Spot FX) 的区分控制
在 `lof_config.yaml` 配置文件中通过 `rate_type` 字段进行控制：
1.  **在岸价基金 (Spot Rate 基金)**：
    对于配置了 `rate_type: spot` 的基金（如富国标普石油 513350、嘉实标普生物 159502），这些基金使用在岸人民币汇率对净值进行计价。
    *   实时计算时，系统直接从新浪获取实时 USD/CNY 在岸价作为“当前汇率”。
    *   为保障盘中高频刷新的可靠性，在岸价缓存 TTL 设定为 15 秒。
    *   **安全防降级机制**：若新浪在岸价接口超时或返回无效数据，系统将**放弃并跳过该基金的估值**，绝对禁止使用央行中间价进行降级兜底，以防发生错误的折溢价计算造成实盘亏损。
2.  **普通基金**：
    对于未配置 `rate_type: spot` 的基金，其“当前汇率”和“基准汇率”统一使用央行每日公布的人民币汇率中间价 `usd_cny_mid`。由于中间价在交易日内是不变的，系统在盘中直接读取内存快照以降低数据库 IO 开销。

## 4. 美国假期回溯与时差处理机制

QDII 欧美基金在盘中估值时面临严重的交易时差和节假日不对称问题：
*   **交易时差**：A 股交易时间（9:30-15:00）美股已收盘，因此 $T$ 日 A 股盘中所使用的“ETF 实时价”实际上是美国前一天（即 $T-1$ 日）的收盘价。
*   **假期不对称**：若 $T-1$ 日是美国假期（美股休市），而 $T$ 日 A 股正常交易。此时虽然系统获取了最新的基金净值 $\text{NAV}_{T-1}$（由于基金公司估值日为 $T-1$ 日），但由于美股休市，数据库中没有这一天任何美股 ETF 的价格数据。

### 4.1 假期回溯算法
在 `dynamic_valuation.py` 中，针对包含美股的分类（黄金原油、QDII欧美等），系统在加载 $T-1$ 日基准数据时，会通过 SQL 校验 `usa_etf_daily_prices` 在此基准日是否包含有效的 ETF 价格。
*   如果 `etf_count == 0`，代表最新净值日 $T-1$ 美国处于休市状态。
*   系统会**向前自动回溯**，定位到最近一个既有基金净值、又有美股 ETF 历史收盘价的有效交易日作为实际的基准日，获取对应的基准价格与基准汇率。
*   该机制完美避免了因基准日底层数据缺失而导致盘中动态估值失效、出现归零脏数据或误报警的情况。

## 5. Level 2 灾备兜底机制与自研净值映射

为了避免估值系统过度依赖 Woody API 提供的 `hedge` 因子，系统底层实现了健壮的 **Level 2 自研灾备机制**。当 Woody API 出现故障、网络阻断或未能提供 `hedge` 数据时，无论静态估值还是盘中实时估值计算器，都会无缝接管并实现数学等价的兜底。

### 5.1 兜底原理：COALESCE 与矩阵公式降级
当魔法公式由于缺少 `hedge` 无法执行时，代码会自动降级执行 **矩阵公式 (Basket Valuation)**。矩阵公式需要获取底层 ETF 的 $T-1$ 日基准分母。

在 `static_valuation.py` 和 `dynamic_valuation.py` 中，查询基准数据的核心 SQL 语句均为：
```sql
SELECT symbol, COALESCE(NULLIF(netvalue, 0), price) as price FROM usa_etf_daily_prices
```
这句 SQL 是兜底机制的灵魂：
1. 它优先提取我们通过自研爬虫 (`US_etf_nav.py`) 独立采集落库的 **`netvalue`**（ETF 官方日终净值）。
2. 只有当 `netvalue` 缺失时，才会退化使用 `price`（收盘价）。

### 5.2 灾备的数学等价性
Woody 的 `hedge` 本质上是由公式推导出的乘数比例：
$$\text{Hedge} \approx \frac{\text{美股ETF净值} \times \text{汇率}}{\text{基金净值} \times \text{仓位}}$$

而当系统降级使用矩阵公式，并将分母精准地替换为自研爬取的 `netvalue` 时，我们计算所使用的波动率基准与 Woody 计算 `hedge` 所用的基准在数学上是 **100% 等价的**。

### 5.3 适用范围
*   **已覆盖兜底的标的**：`XOP` (华宝油气等)、`SPY` (标普500 LOF)、`QQQ` (纳指 LOF) 等。这些代码已被纳入每日定时的 `US_etf_nav.py` 爬虫列表，确保每日 `netvalue` 入库，享有完美的 Level 2 灾备能力。
*   **不适用兜底的标的**：如日经 QDII 所追踪的 `NKY`（日经225期货主连），由于期货本身不存在“净值 (NAV)”的金融概念，因此无法通过爬取净值来实现等价反推。此类基金在失去 Woody 支持时，只能纯粹依赖指数涨跌幅或粗糙的收盘价进行降级。
