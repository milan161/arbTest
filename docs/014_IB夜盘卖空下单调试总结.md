## 二、错误清单（按时间顺序）

### 错误 1：Error 10268 —— 不支持"EtradeOnly"委托单属性

**现象：**
```
[4;2;2;10268;不支持"EtradeOnly"委托单属性。]
```

**原因：** `ibapi` 的 `Order` 类包含 `eTradeOnly` 和 `firmQuoteOnly` 属性。在某些 TWS/Gateway 版本中，这两个属性默认非 `False`，导致拒单。

**修复：** 在 `place_us_order()` 中显式设置为 `False`：
```python
order.eTradeOnly = False
order.firmQuoteOnly = False
```

**位置：** `arbcore/fetchers/ib_reader.py`，`place_us_order()` 方法中设置完 `outsideRth` 之后。

---

### 错误 2：orderId = 0 导致订单被拒

**现象：** TWS 日志显示 `orderId=0`，ETradeOnly 拒单后出现 orderId=0 或异常订单 ID。

**原因：** `reconnect()` 和 `connect_to_ib()` 没有重置 `next_order_id = None`，导致重连后 `reqIds(-1)` 的 `nextValidId` 回调可能不触发，订单 ID 为 None 或 0。

**修复：**
- `reconnect()` 开头加 `self.next_order_id = None`
- `connect_to_ib()` 开头加 `self.next_order_id = None`

**位置：** `arbcore/fetchers/ib_reader.py`

---

### 错误 3：连接成功但行情不推送（"等待数据"）

**现象：** IB 按钮绿灯，但前端盘口显示"等待数据"。

**原因：** `start_polling()` 只在第一次 `get_price()` 调用时懒加载触发（`market_data_service.py` 第 144-145 行）。如果前端请求没触发 get_price，或者 get_price 返回 None 就去兜底富途了，轮询线程不会启动。

**修复：** 在 `reconnect()` 连接成功后显式调用 `self.start_polling()`：
```python
if self.connect_to_ib():
    self.start_polling()
```

**位置：** `arbcore/fetchers/ib_reader.py`，`reconnect()` 方法。

---

### 错误 4：日志刷屏 —— "⏳ IB正在获取XXX，请稍后..."

**现象：** 多条日志反复出现：
```
⏳ IB正在获取GLD，请稍后...
⏳ IB正在获取USO，请稍后...
```

**原因：** 每次前端轮询调用 `get_price()` 时，只要 `ib_reader.prices` 中还没有该 symbol，就会打一次日志。87 只基金 × 15 秒轮询 = 大量刷屏。

**修复：** 添加限频机制，每个 symbol 每 30 秒只打一次日志：
```python
now = time.time()
if not hasattr(self, '_ib_wait_log_time'):
    self._ib_wait_log_time = {}
last_log = self._ib_wait_log_time.get(symbol, 0)
if now - last_log > 30:
    logger.info(f"⏳ IB正在获取{symbol}，请稍后...")
    self._ib_wait_log_time[symbol] = now
```

**位置：** `ArbDashboard/backend/services/market_data_service.py`，`get_price()` 方法。

---

### 错误 5：IB 按钮显示绿色但实际未连接（卡死无法点击）

**现象：**
- 前端侧边栏 IB 按钮为绿色（success），点击无反应
- IB Gateway 无新 Client ID 窗口，后端日志无 `[IBReader] 尝试连接` 消息
- 但 `get_active_source_names()` 返回了 `"IB (Ready)"`，因为 `ib_reader.connected` 为 `True`

**原因：**
`ib_reader.connected` 标志位被错误置为 `True`（可能来自历史重启、异常状态残留），而前端 `reconnectWithGuard(hasIb, 'IB', reconnectIB)` 在 `hasIb=true` 时直接 return，导致用户无法通过按钮重新发起连接。

**Root Cause 链路：**
```
ib_reader.connected=True（误标）
  → get_active_source_names() 返回 "IB (Ready)"
  → 前端 hasIb = true
  → reconnectWithGuard(isConnected=true) → return  ← 点击被吞
  → 用户无法重连
```

**修复：** 移除 IB 按钮的 `reconnectWithGuard`，始终允许点击。后端 `reconnect()` 内部使用 `EClient.isConnected()`（真实 socket 检测）而非 `self.connected` 标记，所以即使已连接也不会有副作用。

```vue
<!-- 修复前 -->
@click="reconnectWithGuard(hasIb, 'IB', reconnectIB)"
:style="{ cursor: hasIb ? 'default' : 'pointer' }"

<!-- 修复后 -->
@click="reconnectIB()"
:style="{ cursor: 'pointer' }"
```

**位置：** `ArbDashboard/frontend/src/layouts/MainLayout.vue`，IB 按钮的 `n-tag` 组件。

**诊断方法：** 后端日志无 `[IBReader] 尝试连接` + IB Gateway 无新 Client ID + 前端按钮绿色不可点击 = 命中此 Bug。

## 三、下单流程备忘

### 3.1 夜盘交易所路由

```python
if self.is_us_night_session():
    contract.exchange = "OVERNIGHT"
else:
    contract.exchange = "SMART"
```

**夜盘判断逻辑（北京时间）：**
- 夏令时（3-11月）：08:00 - 15:50
- 冬令时（11-3月）：09:00 - 16:50
- 仅周一到周五

### 3.2 Primary Exchange 规则

核心标的（QQQ, SPY, GLD, USO, XOP, XBI, SLV）：
- **夜盘（OVERNIGHT）**：必须**移除** `primaryExchange`，否则 Gateway 的 Sec-def 断连时容易导致 Error 201 废单
- **非夜盘（SMART）**：需要设置 `primaryExchange`

```python
primary_map = {"QQQ": "NASDAQ", "SPY": "ARCA", "GLD": "ARCA", ...}
if symbol in primary_map and not self.is_us_night_session():
    contract.primaryExchange = primary_map[symbol]
```

### 3.3 卖空指令

```python
if action == "SELL":
    order.shortSaleSlot = 1
```

注意：XOP 在部分时段不可卖空（Error 201: "该合约不可卖空"），这是 IB 侧限制，不是代码问题。

### 3.4 警告 2109（良性）

```
委托单事件警告：根据委托单类型和目的地，属性"常规交易时间以外"被忽略。现在正在处理PlaceOrder。
```

OVERNIGHT 交易所本身支持盘外交易，`outsideRth=True` 标记是多余的，但无害。**这个警告可以忽略。**

## 四、TWS 日志阅读指南

TWS 日志位于 TWS/Gateway 的 `ibgateway/jts/logs/` 目录下（而非项目目录）。

关键消息格式：

| 方向 | 格式 | 含义 |
|------|------|------|
| `<-` | `[3;2;0;XOP;...]` | OUT: 下单指令。3=版本, 2=消息类型, 0=orderId |
| `->` | `[4;2;2;10268;...]` | IN: 错误响应。4=版本, 2=消息类型, 2=reqId/orderId, 10268=错误码 |
| `->` | `[1;6;reqId;tickType;price;size;...]` | IN: 实时行情推送。1=tickPrice, 2=tickSize |
| `->` | `[17;reqId;...]` | IN: 历史数据响应 |
| `<-` | `[20;reqId;...]` | OUT: 历史数据请求 |

**常见错误码对照：**

| 错误码 | 含义 | 处理方式 |
|--------|------|---------|
| 10268 | EtradeOnly 属性不支持 | 设置 `order.eTradeOnly = False` |
| 2157 | Sec-def 数据场断连 | 忽略，不影响行情和下单 |
| 2104/2106 | 数据场连接正常 | 正常信息 |
| 2107 | 历史数据场未激活 | 正常信息 |
| 2109 | OutsideRth 被忽略 | 良性警告 |
| 201 | 合约不可卖空 | IB 侧限制，换标的 |
| 502/1100 | 连接断开 | 触发重连 |

## 五、AI 修改红线（永久锁定）

以下是对 `ib_reader.py` 的**永久规则**：

1. **`order.eTradeOnly = False` 不允许删除或注释掉** — 删掉 = Error 10268 拒单
2. **`next_order_id = None` 必须在 `reconnect()` 和 `connect_to_ib()` 开头重置** — 否则 orderId=0 废单
3. **`start_polling()` 必须在 `reconnect()` 成功连接后调用** — 否则行情不推送
4. **`exchange="OVERNIGHT"` 夜盘路由不得擅自修改为 SMART** — 参考测试脚本已验证
5. **`primaryExchange` 在夜盘必须留空** — 否则 Sec-def 断连时废单
6. **不允许用 `primary_map` 以外的符号覆盖逻辑** — 新增核心标的需确认 IB 支持
7. **`reconnectWithGuard` 不得用于 IB 按钮（MainLayout.vue）** — 必须直接用 `@click="reconnectIB()"`，因为 `ib_reader.connected` 可能被误标为 True 导致按钮卡死无法点击。后端 `reconnect()` 用真实 socket 检测 `EClient.isConnected()` 做判断，直接调用始终安全。

## 七、关闭连接的生命周期（优雅断开 vs IB Gateway UI 行为）

### 7.1 问题现象

每次重启后端，IB Gateway 的 API 连接面板都会多一行 client 记录。即使程序调用了 `disconnect()`，旧条目也不会消失。

### 7.2 原因

**IB Gateway 设计如此。** Gateway 的 API 连接面板是一个**连接日志**，每条连接记录永久保留，断开后只标记为 "Disconnected"，不会自动删除。每次新连接（即使是同一个程序重启）追加一个新行，这是正常行为。

### 7.3 之前存在的 Bug：shutdown 时 polling 线程抢重连

**现象：** 后端 shutdown 时调用了 `disconnect_from_ib()` 关闭 socket，但 `_polling_loop` 线程还在跑（`self.running` 仍为 True），循环检测到 `self.connected = False` 后立即调用 `connect_to_ib()` 重新连接，导致 Gateway 上同时残留两个 Connected 条目。

**影响链路：**
```
shutdown 
  → disconnect_from_ib()         # 关 socket，设 connected=False
  → polling 线程还在跑            # 未 stop！
  → 检测到 not self.connected
  → connect_to_ib()              # 抢重连！新连接建立
  → 进程退出                      # 第二个连接变成异常断开
```

**修复（2026-07-13）：**

1. **`main.py` shutdown 段：先 `stop_polling()` 再 `disconnect_from_ib()`**
   ```python
   if market_data_service and market_data_service.ib_reader:
       market_data_service.ib_reader.stop_polling()       # 先停轮询线程
       market_data_service.ib_reader.disconnect_from_ib() # 再断开连接
   ```

2. **`ib_reader.py` `disconnect_from_ib()`：增加 TCP FIN 传播延迟**
   ```python
   def disconnect_from_ib(self):
       if self.isConnected():
           self.disconnect()
           self.connected = False
           time.sleep(1)  # 确保 TCP FIN 包到达 Gateway
           logger.info("[IB] 已断开与 Gateway 的连接")
   ```

### 7.4 固定 vs 随机 Client ID

```python
# 当前行为：每次启动用随机 client ID
self.client_id = random.randint(1000, 9999)
```

随机 ID 会加重 Gateway 面板堆积——每次重启视为一个全新的客户端。改成固定 ID（如 `1`）可复用同一行条目，但这只是 UI 层面的优化，不影响功能。**即使固定 ID，断开后该条目仍显示 "Disconnected"，这是 Gateway 面板的固有行为。**

### 7.5 完整的连接生命周期

```
程序启动 → reconnect() → start_polling()
                            ├─ polling 循环（每 5 秒）
                            │   ├─ 检查夜盘时段
                            │   ├─ 维持 reqMktData 长连接订阅
                            │   └─ 看门狗（60 秒无 tick 则安全快照兜底）
                            │
程序关闭 → lifespan yield → stop_polling()   ← 先停止轮询线程
                           → disconnect_from_ib()
                               ├─ EClient.disconnect() → 关闭 socket
                               ├─ time.sleep(1)        → 等 TCP FIN 发出
                               └─ logger.info("已断开")
```

### 7.6 AI 红线补充

8. **`stop_polling()` 必须在 `disconnect_from_ib()` 之前调用** — 否则 polling 线程会抢重连
9. **`disconnect_from_ib()` 内必须有 `time.sleep(1)`** — 否则进程退出太快，TCP FIN 未到达 Gateway
