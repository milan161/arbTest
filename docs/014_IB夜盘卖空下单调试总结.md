# IB 夜盘交易与行情调试总结

> 编写日期：2026-07-02 | 最后更新：2026-07-17
> 核心目的：记录 IB API 夜盘行情订阅和下单的常见坑和修复方案，防止后续 AI 反复踩坑

---

## ⚠️ 任何 AI 修改 ib_reader.py 前必须阅读（2026-07-17 强制执行）

### 关于 OVERNIGHT 交易所的铁律

1. **美股夜盘（OVERNIGHT 时段）有完整的 Level 1 实时行情（Bid/Ask/Last）。** GLD、XOP、USO、QQQ、SPY 等核心套利标的在 Overnight Session 流动性良好，有真实的买一价/卖一价。**AI 不得自作聪明猜测"美股夜盘没有盘口数据"、"夜盘没有流动性 BID/ASK"或擅自引入降级逻辑。** 这是 AI 反复犯的严重错误——总是假设夜盘只有成交价没有盘口，擅自改成快照模式或 SMART 兜底，导致买一价/卖一价丢失。**OVERNIGHT 时段有完整的买一/卖一/成交量数据，与日盘一样完整。**

2. **任何时候，行情订阅和下单路由只使用 `exchange="OVERNIGHT"`。绝对不使用 SMART 路由。** 该程序只在美股夜盘时段运行，不参与日盘交易。任何 AI 不得将 `exchange` 改为 SMART 或做动态切换（夜盘 OVERNIGHT / 日盘 SMART）。

3. **`primaryExchange` 在夜盘必须留空。** OVERNIGHT 路由不需要 `primaryExchange`，加了会导致 Sec-def 断连时的 Error 201 废单。

4. 如果前端外盘盘口显示"等待数据"但 IB 按钮是绿色：
   - 先检查后端控制台是否有 `已发起 GLD 夜盘长连接订阅`
   - 如果有，等 10-30 秒让 IB 推送第一个 tick
   - 如果长时间无数据 → **重启 IB Gateway**，这是最有效的修复手段
   - ~~AI 不得先入为主修改 exchange 或加 SMART 降级兜底~~

### 关于 `reqMktData` 行情订阅

`reqMktData(req_id, contract, genericTickList="", snapshot=False, regulatorySnapshot=False, marketDataOptions=[])` 使用空 `genericTickList` 即可获取完整的 Level 1 数据（Bid=1, Ask=2, Last=4, BidSize=0, AskSize=3）。OVERNIGHT 交易所会正常推送买一价和卖一价，不需要额外参数。

**⚠️ 严禁将 `snapshot` 改为 `True`。** 快照模式（`snapshot=True`）只返回一次成交价（Last），不推送实时 bid/ask，前端会显示"无买一价"或 bid=0。2026-07-17 AI 修改导致没有买一价/卖一价的根因就是这一步——要么改了 `snapshot=True`，要么在回调中错误过滤了 bid/ask 的 tickType。恢复 git 版本 `59091c8`（使用长连接 `snapshot=False`）后行情恢复正常。

## 二、错误清单（按时间顺序）

### 错误 1：Error 10268 —— 不支持"EtradeOnly"委托单属性

**现象：**
```
[4;2;2;10268;不支持"EtradeOnly"委托单属性。]
```

**原因：** `ibapi` 的 `Order` 类包含 `eTradeOnly` 和 `firmQuoteOnly` 属性。在某些 TWS/Gateway 版本中，这两个属性默认非 `False`，导致拒单。

**修复：** 在 `place_us_order()` 中显式设置为 `False`：
```python
order.eTradeOnly = False
order.firmQuoteOnly = False
```

**位置：** `arbcore/fetchers/ib_reader.py`，`place_us_order()` 方法中设置完 `outsideRth` 之后。

---

### 错误 2：orderId = 0 导致订单被拒

**现象：** TWS 日志显示 `orderId=0`，EtradeOnly 拒单后出现 orderId=0 或异常订单 ID。

**原因：** `reconnect()` 和 `connect_to_ib()` 没有重置 `next_order_id = None`，导致重连后 `reqIds(-1)` 的 `nextValidId` 回调可能不触发，订单 ID 为 None 或 0。

**修复：**
- `reconnect()` 开头加 `self.next_order_id = None`
- `connect_to_ib()` 开头加 `self.next_order_id = None`

**位置：** `arbcore/fetchers/ib_reader.py`

---

### 错误 3：连接成功但行情不推送（"等待数据"）

**现象：** IB 按钮绿灯，但前端盘口显示"等待数据"。

**原因：** `start_polling()` 只在第一次 `get_price()` 调用时懒加载触发。如果前端请求没触发 get_price，轮询线程不会启动。

**修复：** 在 `reconnect()` 连接成功后显式调用 `self.start_polling()`：
```python
if self.connect_to_ib():
    self.start_polling()
```

**位置：** `arbcore/fetchers/ib_reader.py`，`reconnect()` 方法。

---

### 错误 4：日志刷屏 —— "⏳ IB正在获取XXX，请稍后..."

**现象：** 多条日志反复出现：
```
⏳ IB正在获取GLD，请稍后...
⏳ IB正在获取USO，请稍后...
```

**原因：** 每次前端轮询调用 `get_price()` 时，只要 `ib_reader.prices` 中还没有该 symbol，就会打一次日志。87 只基金 × 15 秒轮询 = 大量刷屏。

**修复：** 添加限频机制，每个 symbol 每 30 秒只打一次日志。

**位置：** `ArbDashboard/backend/services/market_data_service.py`，`get_price()` 方法。

---

### 错误 5：IB 按钮显示绿色但实际未连接（卡死无法点击）

**现象：**
- 前端侧边栏 IB 按钮为绿色（success），点击无反应
- IB Gateway 无新 Client ID 窗口，后端日志无 `[IBReader] 尝试连接` 消息
- 但 `get_active_source_names()` 返回了 `"IB (Ready)"`，因为 `ib_reader.connected` 为 `True`

**原因：**
`ib_reader.connected` 标志位被错误置为 `True`（可能来自历史重启、异常状态残留），而前端 `reconnectWithGuard(hasIb, 'IB', reconnectIB)` 在 `hasIb=true` 时直接 return，导致用户无法通过按钮重新发起连接。

**修复：** 移除 IB 按钮的 `reconnectWithGuard`，始终允许点击。后端 `reconnect()` 内部使用 `EClient.isConnected()`（真实 socket 检测）而非 `self.connected` 标记，所以即使已连接也不会有副作用。

```vue
<!-- 修复后 -->
@click="reconnectIB()"
:style="{ cursor: 'pointer' }"
```

**位置：** `ArbDashboard/frontend/src/layouts/MainLayout.vue`，IB 按钮的 `n-tag` 组件。

**诊断方法：** 后端日志无 `[IBReader] 尝试连接` + IB Gateway 无新 Client ID + 前端按钮绿色不可点击 = 命中此 Bug。

## 三、下单流程备忘

### 3.1 交易所路由（永久固定为 OVERNIGHT）

本程序只在美股夜盘时段运行，**永远使用 `exchange="OVERNIGHT"`**：

```python
contract.exchange = "OVERNIGHT"
```

**夜盘时段（北京时间）：**
- 夏令时（3-11月）：08:00 - 15:50
- 冬令时（11-3月）：09:00 - 16:50
- 仅周一到周五

### 3.2 Primary Exchange 规则

OVERNIGHT 路由**必须移除 `primaryExchange`**，否则 Gateway 的 Sec-def 断连时容易导致 Error 201 废单：

```python
# OVERNIGHT 不需要 primaryExchange
# 不要设置 contract.primaryExchange
```

### 3.3 卖空指令

```python
if action == "SELL":
    order.shortSaleSlot = 1
```

注意：XOP 在部分时段不可卖空（Error 201: "该合约不可卖空"），这是 IB 侧限制，不是代码问题。

### 3.4 警告 2109（良性）

```
委托单事件警告：根据委托单类型和目的地，属性"常规交易时间以外"被忽略。现在正在处理PlaceOrder。
```

OVERNIGHT 交易所本身支持盘外交易，`outsideRth=True` 标记是多余的，但无害。**这个警告可以忽略。**

## 四、TWS 日志阅读指南

TWS 日志位于 TWS/Gateway 的 `ibgateway/jts/logs/` 目录下（而非项目目录）。

关键消息格式：

| 方向 | 格式 | 含义 |
|------|------|------|
| `<-` | `[3;2;0;XOP;...]` | OUT: 下单指令。3=版本, 2=消息类型, 0=orderId |
| `->` | `[4;2;2;10268;...]` | IN: 错误响应。4=版本, 2=消息类型, 2=reqId/orderId, 10268=错误码 |
| `->` | `[1;6;reqId;tickType;price;size;...]` | IN: 实时行情推送。1=tickPrice, 2=tickSize |
| `->` | `[17;reqId;...]` | IN: 历史数据响应 |
| `<-` | `[20;reqId;...]` | OUT: 历史数据请求 |

**常见错误码对照：**

| 错误码 | 含义 | 处理方式 |
|--------|------|---------|
| 10268 | EtradeOnly 属性不支持 | 设置 `order.eTradeOnly = False` |
| 2157 | Sec-def 数据场断连 | 忽略，不影响行情和下单 |
| 2104/2106 | 数据场连接正常 | 正常信息 |
| 2107 | 历史数据场未激活 | 正常信息 |
| 2109 | OutsideRth 被忽略 | 良性警告 |
| 201 | 合约不可卖空 | IB 侧限制，换标的 |
| 502/1100 | 连接断开 | 触发重连 |

## 五、AI 修改红线（永久锁定，AI 每改 ib_reader.py 前必须逐条核查）

以下是对 `ib_reader.py` 的**永久规则**：

| # | 规则 | 后果 |
|---|------|------|
| **1** | **`exchange="OVERNIGHT"` 永远不变。严禁改成 SMART 或做动态切换** | SMART 无夜盘盘口，废单 |
| **2** | **`order.eTradeOnly = False` 不得删除或注释掉** | Error 10268 拒单 |
| **3** | **`next_order_id = None` 必须在 `reconnect()` 和 `connect_to_ib()` 开头重置** | orderId=0 废单 |
| **4** | **`start_polling()` 必须在 `reconnect()` 成功后调用** | 行情不推送 |
| **5** | **`primaryExchange` 在夜盘不得设置，必须留空** | Sec-def 断连时 Error 201 废单 |
| **6** | **`stop_polling()` 必须在 `disconnect_from_ib()` 之前调用** | polling 线程抢重连 |
| **7** | **`disconnect_from_ib()` 内必须有 `time.sleep(1)`** | TCP FIN 未到达 Gateway |
| **8** | **`reconnectWithGuard` 不得用于 IB 按钮（MainLayout.vue）** | 按钮卡死无法重连 |
| **9** | **断连时必须清除 `self.prices = {}` 和 `self.last_update_time = None`** | 前端误判为 Ready |
| **10** | **`reqMktData` 的 `snapshot` 参数必须为 `False`（长连接流式订阅）。严禁改成 `snapshot=True` 或使用 `reqMktSnapshot`** | snapshot=True 只返回成交价，无 bid/ask，导致所有套利策略失效 |
| **11** | **`tickPrice`/`tickSize` 回调内禁止对 tickType 做任何过滤。凡是 tickType=1/2/4(bid/ask/last) 和 0/3(bid_size/ask_size) 都必须保留。** 禁止加 `if tickType in [...]` 的白名单过滤，禁止只保留 `tickType=4`（成交价）丢弃 1/2（盘口） | 过滤掉 bid/ask 后只余成交价，前端无盘口数据 |

## 七、行情订阅架构概要

```
IB Gateway
  → reqMktData(req_id, Contract(exchange="OVERNIGHT"), genericTickList="", snapshot=False)
  → tickPrice(tickType=1, price) → prices[sym].bid = price     # 买一价
  → tickPrice(tickType=2, price) → prices[sym].ask = price     # 卖一价
  → tickPrice(tickType=4, price) → prices[sym].last = price    # 最近成交价
  → tickSize(tickType=0, size)   → prices[sym].bid_size = size # 买一量
  → tickSize(tickType=3, size)   → prices[sym].ask_size = size # 卖一量
      ↓
market_data_service.get_price()
  → bid>0 且 bid≠ask → return source="IB"
      ↓
前端 foreignSource 显示 "IB"，买一/卖一显示真实盘口价差
```

## 八、关闭连接的生命周期

### 8.1 Gateway 面板行为（无需担心）

每次重启后端，IB Gateway 的 API 连接面板都会多一行 client 记录。**这是 Gateway 的设计**——每条连接记录永久保留，断开后只标记为 "Disconnected"，不会自动删除。每次新连接（即使是同一个程序重启）追加一个新行，这是正常行为，**不影响功能**。

### 8.2 完整连接生命周期

```
程序启动 → reconnect() → start_polling()
                            ├─ polling 循环（每 5 秒）
                            │   ├─ 检查夜盘时段
                            │   ├─ 维持 reqMktData 长连接订阅
                            │   └─ 看门狗（60 秒无 tick 则安全快照兜底）
                            │
程序关闭 → lifespan yield → stop_polling()   ← 先停止轮询线程
                           → disconnect_from_ib()
                               ├─ EClient.disconnect() → 关闭 socket
                               ├─ time.sleep(1)        → 等 TCP FIN 发出
                               └─ logger.info("已断开")
```
