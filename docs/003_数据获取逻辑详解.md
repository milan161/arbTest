# 数据获取逻辑详解

## 📅 更新时间
**2026-04-01**

---

## 🎯 总体原则

系统的数据获取逻辑遵循以下核心原则：

1. **避免重复爬取**：优先检查本地CSV文件中是否已有需要的数据
2. **按时间窗口判断**：根据不同市场的交易时间，判断是否需要获取当日数据
3. **日志清晰**：统一使用"表格读取"、"woody爬取"、"新浪爬取"、"东财爬取"等术语

---

## 📊 数据获取流程图

```
程序启动
   ↓
第一步：检查T-1日历史数据完整性
   ↓
   ├─ 完整 → 无需爬取历史数据
   └─ 不完整 → 计算缺失日期，准备爬取
   ↓
第二步：检查T日特定ETF数据
   ↓
   ├─ 14:35之后 → 检查GLD-JP/USO-JP
   ├─ 16:35之后 → 检查USO-HK
   └─ 其他时间 → 跳过T日特定ETF检查
   ↓
第三步：执行数据获取
   ↓
   ├─ 表格读取：已有数据，直接从CSV读取
   ├─ woody爬取：从Woody网页获取
   ├─ 新浪爬取：从新浪API获取
   └─ 东财爬取：从东财API获取
```

---

## 🔍 第一步：检查T-1日历史数据

### 目的
确保所有ETF和指数在T-1日（上一个交易日）都有完整的数据。

### 逻辑
1. 遍历所有需要检查的ETF和指数
2. 检查每个标的在T-1日是否有数据
3. 如果所有标的都有T-1日数据 → 历史数据完整
4. 如果有标的缺少T-1日数据 → 计算从最近数据日期到T-1日的所有缺失交易日

### 输出示例
```
【第一步】检查T-1日(2026-03-31)的历史数据是否完整
GLD：T-1日(2026-03-31)数据已存在
^GLD-JP：T-1日(2026-03-31)数据已存在
^GLD-EU：T-1日(2026-03-31)数据已存在
...
✅ T-1日历史数据完整，无需爬取历史数据
```

---

## ⏰ 第二步：检查T日特定ETF数据

### 目的
根据不同市场的交易时间，判断是否需要获取当日（T日）的特定ETF数据。

### 时间窗口判断（以北京时间为准）

#### 1. GLD-JP 和 USO-JP（日本市场）
- **检查时间**：北京时间 14:35 之后
- **原因**：日本股票交易在北京时间14:00结束，14:35左右收盘价出来
- **逻辑**：
  - 如果当前时间 < 14:35 → 不检查，不爬取
  - 如果当前时间 ≥ 14:35 → 检查CSV中是否已有T日数据
    - 已有数据 → 跳过
    - 没有数据 → 准备爬取

#### 2. USO-HK（香港市场）
- **检查时间**：北京时间 16:35 之后
- **原因**：香港股票交易在北京时间16:00结束，16:35左右收盘价出来
- **逻辑**：
  - 如果当前时间 < 16:35 → 不检查，不爬取
  - 如果当前时间 ≥ 16:35 → 检查CSV中是否已有T日数据
    - 已有数据 → 跳过
    - 没有数据 → 准备爬取

#### 3. 其他ETF和指数（美股/欧洲市场）
- **检查策略**：不检查T日数据
- **原因**：
  - 美股交易时间在北京时间晚上，当日数据要到次日才会完整
  - 避免在非交易时段误判数据缺失
  - 只需要确保T-1日数据完整即可

### 输出示例
```
【第二步】检查是否需要爬取T日(2026-04-01)的特定ETF数据
当前时间 < 14:35，不检查GLD-JP和USO-JP的T日数据
当前时间 < 16:35，不检查USO-HK的T日数据
```

## 📥 第三步：执行数据获取

### 数据来源统一用词

| 数据来源 | 术语 | 说明 |
|---------|------|------|
| 本地CSV文件 | **表格读取** | 从 `GLD_USO_basic_data.csv` 等本地文件读取 |
| Woody网页 | **woody爬取** | 从 palmmicro/woody 网站爬取数据 |
| 新浪网页/API | **新浪爬取** | 从新浪财经网站或API获取数据 |
| 东财网页/API | **东财爬取** | 从东方财富网站或API获取数据 |

### 执行逻辑

1. **合并需要爬取的ETF列表**：
   - 历史数据缺失的ETF
   - T日特定ETF（GLD-JP/USO-JP/USO-HK）
   - 去重

2. **逐个处理ETF**：
   - 判断是否需要爬取（是否在需要爬取的列表中，或是否强制更新）
   - 不需要爬取 → 表格读取，从本地CSV读取
   - 需要爬取 → 执行网络获取（新浪爬取或woody爬取）

3. **强制更新机制**：
   - 通过环境变量 `FORCE_WOODY_UPDATE=1` 启用强制更新
   - 启用后，即使数据已存在也会重新爬取

### 输出示例
```
开始智能获取锚点数据...
✅ GLD 表格读取，已存在数据，跳过网络请求。
✅ ^GLD-JP 表格读取，已存在数据，跳过网络请求。
...
```

## 💰 期货数据每日一次读取

### 期货校准值

- **保存字段**：`future_gold_calibration`（黄金校准值）、`future_oil_calibration`（原油校准值）
- **更新标记**：`last_calibration_date`
- **逻辑**：
  - 每次运行时检查 `last_calibration_date` 是否等于今天
  - 如果等于今天 → 跳过获取，直接使用配置文件中保存的值
  - 如果不等于今天 → 从Woody网页获取，并保存到配置文件

### 期货结算价

- **品种**：GC（黄金）、CL（原油）、NQ（纳斯达克100）、ES（标普500）
- **更新标记**：`last_futures_settlement_date`
- **逻辑**：
  - 每次运行时检查 `last_futures_settlement_date` 是否等于今天
  - 如果等于今天 → 跳过获取
  - 如果不等于今天 → 从新浪API获取，并保存更新状态到配置文件

### 配置文件示例（lof_config.yaml）
```yaml
future_gold_calibration: 10.9611
future_oil_calibration: 0.8014
last_calibration_date: '2026-04-01'
last_futures_settlement_date: '2026-04-01'
```

### 输出示例
```
=== 获取校准值 ===
【重要】今天(2026-04-01)已经更新过期货校准值，跳过获取

=== 获取期货结算价数据 ===
【重要】今天(2026-04-01)已经更新过期货结算价，跳过获取
```

## 💡 优化效果

1. **避免重复爬取**：减少对数据源的压力，降低被封风险
2. **避免误判缺失**：根据不同市场的交易时间判断，不会在非交易时段误判数据缺失
3. **日志清晰**：统一用词，易于理解数据来源
4. **性能提升**：优先使用本地数据，减少网络请求

## 📋 LOF012 状态管理详解

### 功能概述

LOF012模块负责生成LOF基金的历史数据，包括净值和收盘价的获取。为了避免重复获取数据，实现了基于状态文件的灵活管理机制。

### 状态管理机制

- 使用统一的`access_status.json`文件记录LOF基金的处理状态
- 每个基金每天只处理一次，避免重复获取数据
- 基于时间和处理状态（是否已获取收盘价、净值）智能决定是否处理基金

### 时间控制逻辑

1. **15:00前**：
   - 只尝试获取净值（收盘价尚未公布）
   - 如果已获取净值，不再重复处理

2. **15:00后**：
   - 可以获取收盘价
   - 一旦获得收盘价，不再重复获取
   - 随时可以尝试获取净值（因为净值发布时间不固定）
   - 一旦获取成功，不再重复爬取

### 状态文件结构

```json
{
  "api": {
    "last_access_date": "2026-04-08",
    "access_times": {
      "915": "2026-04-08 09:16:00",
      "1435": null,
      "1605": null
    }
  },
  "lof": {
    "last_process_date": "2026-04-08",
    "processed_funds": {
      "160719": {
        "has_price": true,
        "has_nav": false
      },
      "161116": {
        "has_price": true,
        "has_nav": false
      }
    }
  }
}
```

### 处理逻辑

```python
# 检查是否需要处理该基金
need_process = False
if current_time < cutoff_time_15:  # 15:00前
    if not fund_status.get('has_nav', False):
        need_process = True
else:  # 15:00后
    if not fund_status.get('has_price', False) or not fund_status.get('has_nav', False):
        need_process = True
```
