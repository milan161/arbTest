## 一、静态估值是什么

静态估值 = 基于 T-1 净值计算的基金理论价值，用于计算静态溢价率：

```
静态溢价率 = (收盘价 / 静态估值 - 1) × 100
```

**关键规则**：静态溢价必须用 **T-1 净值**（前一交易日净值），绝不能用当天净值 bfill 到更早日期。

---

## 二、三种估值算法

| 算法 | 适用基金 | 核心参数 | 公式 |
|------|---------|---------|------|
| **矩阵公式** | 黄金原油、QDII欧美（有 basket 数据的基金） | position + basket 权重 + ETF 价格 + 汇率 | `static_val = base_nav × (1 + position × (Σ(price_ratio × weight) × fx_change - 1))` |
| **魔法公式** | 单一 ETF 跟踪基金（有 hedge 数据） | position + hedge + ETF 价格 + 汇率 | `static_val = base_nav × (1 - position) + (current_price × current_fx) / hedge` |
| **跟踪指数公式** | QDII亚洲、国内LOF（无 Woody 数据） | pos_ratio + 指数比率 + 汇率 | `static_val = prev_nav × (1 + pos_ratio × (idx_ratio × fx_ratio - 1))` |

### 如何判断一只基金用哪种算法？

1. 查 `lof_config.yaml` 中该基金是否有 `valuation_portfolio` 或 `hedging_portfolio`
2. 如果有 `hedging_portfolio` 且有 `hedge` 值 → **魔法公式**
3. 如果有 `valuation_portfolio` → **矩阵公式**
4. 如果两者都没有 → **跟踪指数公式**（step11 处理）

## 三、数据流向

```
lof_config.yaml（基金配置）
    ↓
fund_basket_weights 表（篮子权重，按日期更新）
    ↓
usa_etf_daily_prices 表（ETF 价格）
    ↓
exchange_rate 表（汇率）
    ↓
unified_fund_history 表（净值、收盘价、静态估值、溢价率）
    ↓
fund_service.get_fund_history() → 前端展示
```

### 关键表

| 表 | 用途 | 更新频率 |
|----|------|---------|
| `unified_fund_history` | 存储净值、收盘价、静态估值、溢价率 | 每日 |
| `fund_daily_factors` | 存储 position、hedge、calibration | 每日（Woody API） |
| `fund_basket_weights` | 存储篮子权重（按日期） | 每周/每月 |
| `usa_etf_daily_prices` | 存储美股 ETF 价格 | 每日 |
| `exchange_rate` | 存储汇率 | 每日 |

## 四、计算时机

### 工作日完整流水线（`_run_pipeline()`）

```
step1-2: Woody API 因子
step3: 汇率
step4: 净值更新（东财API）
step4P: 收盘价更新（腾讯API）
step4_fix: 假期价格修复
step5: 美股 ETF 价格（新浪）
step7: 校准值
step8: 期货数据
step9: 份额数据（VPS SFTP）
step10: **静态估值计算** ← 关键步骤
step11: 跟踪指数公式静态估值（覆盖 step10 未处理的基金）
```

### 周末模式（`run()` weekday 5,6）

```
step4: 净值更新（东财API）
step10: **静态估值计算** ← 2026-07-04 新增
```

**⚠️ 重要**：周末模式**不执行** step1-3, 4P, 5, 7-9, 11。因为周末美股休市，不需要更新 ETF 价格和因子。但如果周末更新了净值（东财 API 有时在周末更新），就必须重新计算静态估值。

### nav_only 模式

```
仅执行 step4 净值更新，不计算静态估值。
```

## 五、静态估值计算流程（step10）

### 入口

`daily_updater._step10_calculate_static_valuation()`

### 步骤

1. 读取 `lof_config.yaml`
2. 遍历所有有 `valuation_portfolio` 或 `hedging_portfolio` 的基金
3. 对每只基金调用 `StaticValuationCalculator.process_fund(fund)`
4. `process_fund()` 内部：
   - 查询最近 40 天的基础数据（unified_fund_history + fund_daily_factors + exchange_rate）
   - 对 portfolio 中每个 symbol，merge ETF 价格和 basket 权重
   - 调用 `_calculate_history()` 逐日推演
   - 调用 `_save_results()` 写入数据库

### `_calculate_history()` 核心逻辑

```python
# 1. 按日期 DESC 排序
df = df.sort_values('date', ascending=False)

# 2. 权重 bfill（处理节假日权重缺失）
for item in portfolio:
    sym = item.get('symbol', '').replace('^', '')
    # ⚠️ 必须先处理区域后缀再加 ^
    if any(suffix in sym for suffix in ['-JP', '-EU', '-HK']):
        sym = f"^{sym}"
    w_col = f"{sym}_weight"
    if w_col in df.columns:
        df[w_col] = df[w_col].bfill().fillna(item.get('weight', 0))

# 3. 逐日推演
for i in range(len(df)):
    row = df.iloc[i]
    # 寻找 T-1 基准日（从 i+1 开始找，因为 DESC 排序，i+1 是更早的日期）
    base_row = find_first_valid_base_row(df, i)
    if base_row is None: continue
    
    # 检查基准日跨度，超过 5 天说明长假，跳过
    gap_days = (curr_dt - base_dt).days
    if gap_days > 5: continue
    
    # 调用 _deduce_valuation() 计算
    val = _deduce_valuation(row, base_row, portfolio, primary_sym, fund_config)
    if val:
        df.at[i, 'static_val'] = round(val, 4)
```

### `_deduce_valuation()` 计算逻辑

```python
b_nav = base_row['nav']       # T-1 净值
b_fx = base_row['exchange_rate']  # T-1 汇率
c_fx = row['exchange_rate']       # 当日汇率
position = base_row['position']   # T-1 仓位

# 1. 尝试魔法公式（有 hedge 数据时）
if pd.notna(base_row['hedge']) and base_row['hedge'] > 0 and primary_sym in row:
    c_price = row[primary_sym]
    val = calculate_magic_valuation(b_nav, position, c_price, c_fx, base_row['hedge'])
    if val: return val

# 2. 尝试矩阵公式（有 basket 数据时）
items = []
for p in portfolio:
    sym = p.get('symbol', '').replace('^', '')
    if any(suffix in sym for suffix in ['-JP', '-EU', '-HK']):
        sym = f"^{sym}"
    
    # ⚠️ 列名必须和 merge 时一致（带 ^）
    if sym in row and sym in base_row:
        items.append({
            'current_price': row[sym],
            'base_price': base_row[sym],
            'weight': row.get(f"{sym}_weight", p.get('weight', 0)) / 100.0
        })

return calculate_basket_valuation(b_nav, position, c_fx, b_fx, items)
```

### `calculate_basket_valuation()` 公式

```python
fx_change = current_fx / base_fx
w_change = sum((current_price / base_price) * weight for item in items)
net_ratio = position * (w_change * fx_change - 1.0)
return base_nav * (1.0 + net_ratio)
```

## 六、常见故障排查

### 故障1：static_val 为 None（缺失）

**症状**：历史记录页面 static_val 列为空（显示 `-`）

**排查步骤**：

1. **检查是否是周末**：周末只跑 step4 + step10，如果 step10 没跑，static_val 不会更新
   - 修复：见 [修复2](#修复2周末静态估值缺失)

2. **检查 ETF 价格是否存在**：
   ```sql
   SELECT date, price FROM usa_etf_daily_prices WHERE symbol='GLD' AND date >= '2026-07-01' ORDER BY date DESC;
   ```
   如果 ETF 价格缺失，矩阵公式无法计算。

3. **检查 basket 权重是否存在**：
   ```sql
   SELECT date, underlying_symbol, weight FROM fund_basket_weights WHERE fund_code='160719' ORDER BY date DESC;
   ```
   如果权重缺失，bfill 会用默认值。

4. **检查汇率是否存在**：
   ```sql
   SELECT date, usd_cny_mid FROM exchange_rate WHERE date >= '2026-07-01' ORDER BY date DESC;
   ```

5. **检查 position 是否存在**：
   ```sql
   SELECT date, position FROM fund_daily_factors WHERE fund_code='160719' ORDER BY date DESC;
   ```

6. **检查基准日跨度**：如果 T-1 净值缺失超过 5 天（长假），会跳过计算。

### 故障2：static_val 计算错误

**症状**：static_val 有值但不正确

**排查步骤**：

1. **检查权重列名是否匹配**：这是最常见的 bug 来源
   - portfolio 中的 symbol 格式：`GLD-EU` → merge 时变成 `^GLD-EU`
   - 权重列名：`^GLD-EU_weight`
   - bfill 时必须用 `^GLD-EU_weight`，不能用 `GLD-EU_weight`
   
2. **检查 hedge 值是否正确**：
   ```sql
   SELECT date, hedge FROM fund_daily_factors WHERE fund_code='162411' ORDER BY date DESC;
   ```
   hedge 为 NULL 会导致魔法公式回退到矩阵公式。

3. **检查 T-1 净值是否正确**：静态估值基于 T-1 净值，如果 T-1 nav 错误，static_val 也会错。
   - 确认 `unified_fund_history` 中 T-1 行的 nav 值

4. **手动验证计算**：
   ```python
   # 示例：160719 07-02
   # base_nav = 1.822 (07-01 nav)
   # position = 0.9608
   # fx_change = 6.8088 / 6.8067 = 1.000309
   # GLD ratio = 378.13 / 370.60 = 1.02032, weight = 0.552
   # GLD-EU ratio = 378.50 / 374.79 = 1.01000, weight = 0.448
   # w_change = 1.02032 * 0.552 + 1.01000 * 0.448 = 1.01565
   # net_ratio = 0.9608 * (1.01565 * 1.000309 - 1.0) = 0.01534
   # static_val = 1.822 * (1 + 0.01534) = 1.8499
   ```

### 故障3：周末净值更新后 static_val 不变

**症状**：周末 step4 更新了 nav，但 static_val 还是旧值

**根因**：周末模式之前只跑 step4，不跑 step10。

**修复**：周末模式也执行 `_step10_calculate_static_valuation()`

## 七、符号命名规范（⚠️ 极易出错）

### lof_config.yaml 中的 symbol 格式

**所有区域锚点必须带 `^` 前缀**：

| 正确 | 错误 | 说明 |
|------|------|------|
| `^GLD-EU` | `GLD-EU` | 欧洲锚点黄金 ETF |
| `^USO-HK` | `USO-HK` | 香港锚点原油 ETF |
| `^GLD-JP` | `GLD-JP` | 日本锚点黄金 ETF |
| `GLD` | - | 核心锚点无后缀，不加 `^` |
| `USO` | - | 核心锚点无后缀，不加 `^` |

**规则**：
- 核心锚点（GLD, USO, XOP, SLV, SPY, QQQ, INDA）→ **无前缀**
- 区域变种锚点（带 `-EU`, `-JP`, `-HK` 后缀）→ **必须加 `^` 前缀**

### 数据库中的 symbol 格式

`usa_etf_daily_prices.symbol` 和 `fund_basket_weights.underlying_symbol` 中存储的 symbol **必须和 lof_config.yaml 中的格式一致**。

### 代码中的 symbol 处理

`static_valuation.py` 中的 symbol 处理流程：

```
lof_config.yaml: "GLD-EU"
    ↓ replace('^', '') → "GLD-EU"（去掉已有的 ^，防止重复）
    ↓ 检测区域后缀 → "^GLD-EU"（加上 ^）
    ↓ 用于 merge ETF 价格 → 列名 "^GLD-EU"
    ↓ 用于 merge 权重 → 列名 "^GLD-EU_weight"
    ↓ bfill 时必须用 "^GLD-EU_weight" ← 这是 bug 高发点！
```

**⚠️ 关键**：`_calculate_history()` 中的 bfill 和 `_deduce_valuation()` 中的列名查找**必须使用相同的 symbol 处理逻辑**（先 replace `^`，再检测后缀加 `^`）。
